The Foundation: Database Schema

  • What? Database Initialization Script.
  • Why? To manage data persistence. Creates users and saved_entries tables.
  • How? Runs on startup. Ensures structure consistency across Dev and Cloud.
app.py
def init_db():
  """Initialize SQLite tables if missing"""
  conn = get_db_connection()
  cur = conn.cursor()
  cur.execute("CREATE TABLE IF NOT EXISTS users...")
  cur.execute("CREATE TABLE IF NOT EXISTS saved_entries...")
  conn.commit()

The Logic: Testing Stateful Features

Testing a database is harder than simple math. We used Integration Testing to verify the entire flow.

Test Strategy:

  • Unit Test: Create fake user, hash password, verify login logic.
  • Integration Test: Save IP entry, assert database row count increases.
# test_app.py: Integration Test
def test_save_and_retrieve(client):
  # 1. Login Mock User
  with client.session_transaction() as sess:
    sess['user_id'] = 1

  # 2. Save Data
  resp = client.post('/api/save_entry', json=data)
  assert resp.status_code == 201

  # 3. Verify Retrieval
  resp = client.get('/api/saved_entries')
  assert len(resp.json['entries']) == 1

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
    - name: Install Dependencies
      run: pip install -r requirements.txt

    - name: Run Tests
      env:
        FLASK_SECRET_KEY: ${{ secrets.TEST_KEY }}
      run: pytest

The Sabotage: The Breaking Code

Introducing the Bug

To prove the pipeline works, we wrote a test case that deliberately breaks expectations.

We change the expected status code from 200 (OK) to 500 (Server Error).


Since the app is working correctly (returning 200), this assertion will FAIL the test.

# test_app.py: Intentionally Broken Test
def test_index_page(client):
  resp = client.get('/')

  # SABOTAGE:
  # We changed 200 to 500 to break the build.
  # The app returns 200, so this assertion FAILS.
  assert resp.status_code == 500
